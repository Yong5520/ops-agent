# OpsAgent - PRD

> AI 驱动的 Linux 运维 Agent 桌面客户端

## 1. 项目背景

运维人员在处理主机故障、告警、异常时，通常需要手动 SSH 连接目标主机，逐条执行命令进行排查和修复。这个过程依赖个人经验，效率低且容易遗漏。

现有项目 [ssh-mcp-multi](https://github.com/Yong5520/ssh-mcp-multi) 通过 MCP 协议让 Claude Code 具备了多主机 SSH 操作能力，验证了 AI 辅助运维的可行性。但其依赖 Claude Code CLI 作为宿主，部署和使用门槛较高。

**OpsAgent 的目标是将这一能力产品化**：做成一个独立的 PC 桌面客户端，用户无需安装 Claude Code，只需配置 API Key 或本地模型地址，即可通过自然语言对话的方式，让 AI 自动连接目标 Linux 主机进行诊断、分析和修复操作。

## 2. 核心设计原则

### 2.1 自建 Agent Loop

不依赖 Claude Code CLI 或 Claude Code SDK。自行实现 Agent Loop，通过模型 API（Anthropic / OpenAI / 本地模型）的 tool calling 能力驱动运维操作。可参考 Claude Code 的代码实现思路，但独立构建。

核心 Agent 循环：

```
用户输入 → 构造 System Prompt + 工具定义 → 调用模型 API → 解析 tool_use →
安全过滤 → 用户授权（按模式） → SSH 执行命令 → 结果返回模型 → 循环或回复用户
```

### 2.2 多模型后端支持

不绑定特定模型供应商。支持任何提供 OpenAI 兼容 API 的模型服务：

- Anthropic Claude（通过 `@anthropic-ai/sdk`）
- OpenAI GPT 系列（通过 OpenAI SDK）
- 本地模型服务（Ollama / vLLM / LM Studio 等，通过 OpenAI 兼容 API）
- Azure OpenAI / 其他兼容端点

用户在设置中配置模型供应商、API Key、模型名称即可切换。

### 2.3 安全优先

所有 SSH 命令执行经过安全过滤系统。提供三级安全模式，从严格只读到完全自主，用户按需选择。

### 2.4 完全审计

所有操作（包括被拦截的命令）完整记录，支持查询和导出，满足运维审计要求。

## 3. 功能需求

### 3.1 设置模块

#### 3.1.1 模型配置

管理 AI 模型供应商和连接信息。

- **供应商管理**：支持添加多个模型供应商配置，每个配置包含：
  - 供应商类型（Anthropic / OpenAI / OpenAI 兼容端点）
  - API 端点地址（URL）
  - API Key
  - 模型名称（如 `claude-sonnet-4-6`、`gpt-4o`、`qwen2.5-72b`）
  - 模型能力标签（可选，用于标记适合做规划还是执行）
- **连接测试**：配置保存后可一键验证 API 连通性
- **活跃模型选择**：从已配置的供应商中选择当前使用的模型
- **本地模型支持**：Ollama / vLLM / LM Studio 等本地服务，只需填写本地地址（如 `http://localhost:11434`）即可接入

#### 3.1.2 目标主机配置

管理可通过 SSH 连接的目标 Linux 主机。

- **主机列表**：支持添加、编辑、删除主机，每台主机包含：
  - 自定义名称（别名，如 `n110`、`db-master`）
  - IP 地址或域名
  - SSH 端口（默认 22）
  - 用户名
  - 认证方式（密码 或 SSH 密钥文件）
  - 密码（密码认证时）
  - 密钥文件路径（密钥认证时，支持文件选择器）
  - sudo 密码（可选，用于提权操作）
  - su 密码（可选，用于持久 root shell）
- **连接状态检测**：显示每台主机的在线/离线状态和连接延迟
- **分组管理**：主机可按环境分组（如 prod / staging / dev）
- **导入/导出**：配置可导出为加密文件，方便团队共享（密码加密存储）

#### 3.1.3 危险程度模式

提供三级安全模式，控制 AI 可执行的命令范围。在对话过程中可随时切换。

| 层级 | 名称 | 说明 |
|------|------|------|
| A | **诊断模式 (Sentinel)** | 严格只读。仅允许执行查询、诊断类命令（ls、cat、grep、ps、df、top、systemctl status 等）。任何写入、修改、删除操作均被拦截。用于安全环境下的根因分析。 |
| B | **标准模式 (Operator)** | 允许全部操作，但写入类命令（修改文件、重启服务、安装包等）需用户逐条确认授权后才能执行。授权交互方式参考 Claude Code 的权限确认 UI。 |
| C | **自主模式 (Autopilot)** | AI 可自行决定并执行全部命令，无需人工确认。仅适用于测试环境或用户完全信任的场景。 |

安全过滤系统：
- 复用 ssh-mcp-multi 已有的 18 条危险命令拦截规则作为基础
- 支持用户自定义黑名单/白名单规则
- 命令拦截时记录到审计日志并通知 AI（AI 收到拦截信息后调整方案）
- 防止通过 eval、bash -c、base64 等方式绕过过滤

#### 3.1.4 Skills 配置

Skills 是预定义的运维能力包，为 AI 提供特定领域的知识和操作序列。

- **预设 Skills**（内置）：
  - 系统诊断：磁盘、内存、CPU、网络、进程全面检查
  - 服务管理：nginx、docker、systemd、mysql、redis、java 等常见服务的启停、配置检查、日志查看
  - 日志分析：解析 syslog、nginx access/error log、mysql slow log 等，提取关键错误信息
  - 安全巡检：检查防火墙规则、SSH 配置、用户权限、开放端口等
  - 性能分析：收集 top、iotop、vmstat、sar 等数据并给出优化建议
- **自定义 Skills**：用户可编写自己的 Skill，格式为 Markdown（包含 prompt + 工具链组合）
- **Skill 启用/禁用**：用户可选择当前会话启用哪些 Skills
- Skill 本质上是 **System Prompt 的组成部分** + **推荐工具调用序列**

### 3.2 对话交互模块（Chat UI）

核心交互界面，用户通过自然语言与 AI 交互，AI 通过 SSH 在目标主机上执行操作。

#### 3.2.1 对话基础

- **多轮对话**：保持完整上下文，AI 能理解前序操作的结果并据此决策
- **多主机上下文**：对话中可通过 `@主机名` 指定目标主机（如 `@n110 检查磁盘`），也可在对话开始时选择默认主机
- **Markdown 渲染**：AI 回复支持代码块、表格、列表等格式
- **代码高亮**：命令和输出内容带语法高亮
- **长输出处理**：超过一定长度的命令输出自动折叠，点击展开；支持分页浏览
- **流式输出**：AI 回复逐字/逐块显示，不等待完整生成

#### 3.2.2 授权交互（标准模式下）

当 AI 计划执行写入类命令时，弹出授权确认 UI：

```
┌──────────────────────────────────────────────────────┐
│ 需要授权                                                │
│                                                         │
│ 主机: n110 (10.31.10.110)                               │
│ 安全模式: 标准 (Operator)                                │
│                                                         │
│ 计划执行以下操作:                                        │
│  1. [READ]   df -h                                     │
│  2. [READ]   systemctl status nginx                   │
│  3. [WRITE]  systemctl restart nginx          ← 需授权 │
│  4. [READ]   curl -s -o /dev/null -w "%{http_code}"   │
│     http://localhost                                    │
│                                                         │
│ 操作说明: AI 判断 nginx 服务异常，尝试重启恢复服务       │
│                                                         │
│ [全部批准]    [逐条确认]    [拒绝]                        │
└──────────────────────────────────────────────────────┘
```

授权粒度：
- **全部批准**：一次性批准该批次所有命令
- **逐条确认**：每条命令单独弹出确认，可选择批准/跳过/拒绝
- **拒绝**：拒绝全部操作，AI 收到拒绝信息后需调整方案

#### 3.2.3 执行过程可视化

AI 执行命令时，在对话区域实时显示执行状态：

- 每条命令显示：命令内容、执行状态（等待中 / 执行中 / 成功 / 失败 / 被拦截）、执行耗时
- 按主机分组显示（多主机操作时）
- 失败命令高亮标红，显示错误输出

#### 3.2.4 会话管理

- **新建会话**：创建新的对话上下文
- **会话列表**：显示历史会话（按时间排序），显示摘要（主机、时间、操作概述）
- **恢复会话**：切换到历史会话继续对话（上下文保留）
- **会话导出**：将会话内容导出为 Markdown / PDF

#### 3.2.5 快捷操作

- **对话模板**：预设常见运维场景的快速入口
  - 主机全检（系统全面体检）
  - Nginx 诊断
  - 磁盘空间告警排查
  - 网络连通性检查
  - 服务健康检查
- **快捷命令**：用户可直接输入命令（以 `$` 或 `>` 开头），不经 AI 解析直接执行

### 3.3 日志审计模块

记录所有操作，供事后查询和审计。

#### 3.3.1 记录内容

每条操作记录包含：
- 时间戳
- 会话 ID
- 目标主机（名称 + IP）
- 安全模式
- 命令类型（READ / WRITE / SUDO / BLOCKED）
- 命令内容（完整命令字符串）
- AI 对该命令的描述（AI 为什么执行这条命令）
- 执行耗时
- 返回码（0 / 非零）
- 用户授权状态（自动执行 / 用户批准 / 用户拒绝 / 被安全规则拦截）
- 命令输出摘要（截取关键信息，完整输出可选存储）

#### 3.3.2 查询界面

- **筛选条件**：
  - 按主机筛选
  - 按时间范围筛选
  - 按安全模式筛选
  - 按命令类型筛选
  - 按关键词搜索（搜索命令内容或 AI 描述）
- **展示方式**：
  - 列表视图：每行一条记录
  - 会话视图：按会话分组展示完整操作链
  - 时间线视图：按时间线展示多台主机的操作交错情况
- **导出**：支持导出为 CSV / PDF / Markdown

### 3.4 告警集成（Phase 2）

支持接入外部告警系统，自动触发根因分析。

- **告警源**：
  - Prometheus AlertManager Webhook
  - 钉钉 / 企业微信机器人 Webhook
  - 邮件解析（通过 IMAP/POP3 监听告警邮件）
  - 自定义 Webhook（通用 JSON 格式）
- **处理流程**：
  告警触发 → 解析告警内容 → 自动创建会话 → 选择对应主机 → AI 自动执行诊断 → 生成分析报告 → 推送通知
- **告警-操作全链路追踪**：从告警触发到问题解决的完整操作记录

### 3.5 批量操作（Phase 2）

同时对多台主机执行相同或不同的操作。

- **并行执行**：选中多台主机，执行同一命令，结果汇总对比展示
- **进度面板**：显示每台主机的执行进度（等待 / 执行中 / 完成 / 失败）
- **失败重试**：对失败的主机可批量重试或逐台排查
- **差异对比**：多台主机的执行结果并排对比，高亮差异部分

### 3.6 知识库（Phase 3）

为 AI 提供额外知识，提升诊断和修复能力。

- **运维文档索引**：上传内部运维文档（Markdown / PDF / 纯文本），AI 在分析时可参考
- **历史经验库**：自动从历史会话中提取成功的排查方案，作为 RAG 知识源
- **知识库管理**：添加/删除/更新文档，支持按类别组织

### 3.7 会话回放（Phase 3）

- 完整记录操作链路，支持回放查看
- 可作为运维交接、培训材料
- 支持按时间轴拖拽查看

## 4. Agent Loop 设计要点

### 4.1 工具定义

AI 通过 tool calling 调用以下工具：

| 工具名称 | 功能 | 参数 |
|---------|------|------|
| `exec` | 在指定主机执行普通命令 | host, command, description |
| `sudo_exec` | 在指定主机以 sudo 权限执行命令 | host, command, description |
| `read_file` | 读取目标主机上的文件内容 | host, path, offset?, limit? |
| `write_file` | 写入文件到目标主机 | host, path, content |
| `list_hosts` | 列出所有已配置的主机及其状态 | 无 |
| `run_skill` | 执行预设的运维技能包 | skill_name, host, params? |

### 4.2 System Prompt 结构

```
你是 OpsAgent 运维助手。你将通过 SSH 连接到目标 Linux 主机执行操作。

## 当前主机信息
{当前选中主机的基本信息}

## 可用主机列表
{所有已配置主机的名称、IP、分组}

## 安全模式
{当前安全模式及其限制说明}

## 已启用 Skills
{当前启用的 Skills 描述}

## 安全规则
{危险命令拦截规则，告知 AI 哪些命令不可执行}

## 操作规范
- 每条命令必须提供 description 说明执行目的
- 执行写入操作前会征求用户确认
- 命令被拦截时应调整方案而非重复尝试
- 先诊断后操作，不要盲目执行修复
```

### 4.3 Agent Loop 流程

```
1. 用户输入 → 加入消息历史
2. 构造请求（System Prompt + 消息历史 + 工具定义）
3. 调用模型 API
4. 解析响应：
   a. 如果是纯文本回复 → 展示给用户，等待输入
   b. 如果包含 tool_use → 进入工具执行流程
5. 工具执行流程：
   a. 安全过滤检查 → 拦截则返回 BLOCKED 信息
   b. 用户授权检查（标准模式 + 写入命令）→ 等待用户决定
   c. SSH 执行命令 → 返回结果
   d. 记录审计日志
6. 工具结果加入消息历史 → 回到步骤 2
```

### 4.4 上下文管理

- 会话级上下文：当前对话的完整消息历史
- 当上下文过长时，对早期消息进行摘要压缩（保留关键操作和结论，丢弃冗余输出）
- Skill 描述作为 System Prompt 的一部分注入，不占用对话上下文

## 5. 非功能需求

### 5.1 安全性

- 主机凭据加密存储（使用系统密钥链或 AES 加密）
- API Key 加密存储
- SSH 连接使用加密通道
- 导出配置时密码默认脱敏或加密

### 5.2 可靠性

- SSH 连接池：按需建立连接，自动复用，空闲超时断开
- 断线重连：SSH 连接断开后自动重试
- 命令超时：可配置的单命令执行超时，超时自动终止

### 5.3 性能

- 多主机命令并行执行
- 流式输出提升交互体验
- 本地 SQLite 存储，读写性能有保障

### 5.4 可扩展性

- Skills 系统允许用户和社区贡献新能力
- 工具定义可扩展，后续可添加更多工具（如文件传输、端口转发等）
- 告警源可插拔

## 7. 与 ssh-mcp-multi 的关系

OpsAgent 复用 ssh-mcp-multi 中以下核心能力：

- SSH 连接池实现
- 密码/密钥双认证
- sudo / su 提权
- 18 条危险命令安全过滤规则
- 按主机分文件的审计日志格式

但这些代码需要从 MCP Server 的形态重构为可被桌面应用直接调用的内部模块（不再通过 MCP 协议通信，而是直接函数调用）。
